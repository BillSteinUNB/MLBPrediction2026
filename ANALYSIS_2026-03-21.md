## PART 1: SPEED IMPROVEMENTS (Build Time Reduction)

### 🔴 CRITICAL — The #1 Bottleneck

**`_compute_feature_modules()` in `data_builder.py:828-974` — Sequential day-by-day loop**

This is where the build spends 95%+ of its time. For a single walk-forward window:
- Iterates through **~219 unique game dates** one at a time
- For each date, calls **5 feature modules SEQUENTIALLY**: prepare → offense → pitching → defense → bullpen → baselines
- The last run got stuck at `[build 1/219] ... bullpen` — meaning even day 1 is slow

**Why it's slow per-day:**
- **Offense**: Fetches `team_game_logs()` for every team playing that day (current + prior season)
- **Pitching**: Calls `_fetch_season_start_metrics()` → fetches entire Statcast season via `fetch_statcast_range()` which iterates **day-by-day** creating one parquet file per calendar day (~180 files per season)
- **Defense**: Loops through **ALL 30 teams** to build defensive efficiency (line 363: `for offense_team in TEAM_CODES`) even when only 2-4 teams play
- **Bullpen**: Same Statcast fetching pattern as pitching

**Estimated impact**: 10-50x speedup potential

---

### 🔴 CRITICAL — Statcast Fetch is Day-by-Day

**`statcast_client.py:113-141` — `fetch_statcast_range()`**

```python
for query_date in _iterate_dates(start_date, end_date):
    # ONE parquet file per calendar day
    # ONE API call or file read per day
```

For a full season (April–October), that's **~180 separate parquet file reads** even when cached. The pybaseball `statcast()` call is even worse on first fetch — one HTTP request per day.

**Fix**: Batch into weekly or monthly chunks. Read all cached parquets in one `pd.concat()` call instead of one-at-a-time.

---

### 🟡 HIGH — Parallel Execution Rarely Activates

**`data_builder.py:977-1005` — `_resolve_effective_feature_build_workers()`**

The parallel code path exists (`_compute_feature_modules_parallel` with `ProcessPoolExecutor`), but it returns `workers=1` if ANY of these are true:
- `refresh_raw_data=True`
- Any custom fetcher is provided
- Any lineup data exists

The walk-forward backtest path (`walk_forward.py:816`) calls `build_training_dataset()` which uses default fetchers, so parallel SHOULD activate. But each parallel worker:
1. Creates its own temp SQLite database
2. Seeds the ENTIRE schedule (2882 games) even for a 14-day chunk
3. Re-fetches all team game logs independently (memoization doesn't cross process boundaries)

**Fix**: Pre-fetch and serialize all raw data before spawning workers. Pass data via shared memory or temp parquet files, not re-fetching.

---

### 🟡 HIGH — SQLite EAV Feature Store is Slow

**All feature modules persist via DELETE + INSERT per row**

The features table uses Entity-Attribute-Value layout. For offense alone: 6 metrics × 4 windows × 2 levels (team+lineup) × 2 sides × ~15 games/day = **~1440 individual SQL operations per game date**. Across all modules, ~5000+ row operations per day × 219 days = **1M+ SQL statements per build**.

**Fix**:
- Batch all features into a single DataFrame, write to parquet directly
- Only use SQLite for game schedule and final assembly
- Or use `executemany` with larger batches and fewer commits

---

### 🟡 HIGH — No Checkpoint/Resume

If the build crashes at day 150/219, it restarts from scratch. There's no incremental progress tracking.

**Fix**: Save feature parquet per-chunk (e.g., every 14 days). On restart, skip chunks that already have cached output.

---

### 🟠 MEDIUM — Defense Module Fetches All 30 Teams Unconditionally

**`defense.py:363` — `for offense_team in TEAM_CODES:`**

Even when only 2 teams play on a date, the defense module fetches game logs for all 30 teams to build defensive efficiency histories. This is a ~15x over-fetch.

**Fix**: Only fetch game logs for teams appearing in the schedule for the current build window.

---

### 🟠 MEDIUM — `enable_pybaseball_cache()` Called Redundantly

Every call to `fetch_batting_stats`, `fetch_team_game_logs`, etc. calls `enable_pybaseball_cache()` which re-sets env vars, config, and calls `.save()`. In a full build, this happens hundreds of times.

**Fix**: Call once at process startup, not per-fetch.

---

### 🟠 MEDIUM — Redundant `_prepare_schedule_frame()` Processing

This function is called multiple times on the same data, re-parsing dates, normalizing team codes via `.map()` row-by-row, and re-computing park factors.

**Fix**: Prepare once, cache the result.

---

### 🟢 LOW — tempfile SQLite Per Build

Each build creates a temp SQLite database, which adds disk I/O overhead. The parallel path creates one per chunk.

**Fix**: Use in-memory SQLite (`:memory:`) for builds that don't need persistence, or use DataFrames directly.

## PART 2: MODEL QUALITY IMPROVEMENTS

### 🔴 CRITICAL — Hyperparameter Search is Severely Undercooked

**`xgboost_trainer.py:36-40`**:
```python
DEFAULT_SEARCH_SPACE = {
    "max_depth": [3, 4, 5, 6, 7, 8],        # 6 values
    "n_estimators": [100, 200, 300, 400, 500], # 5 values
    "learning_rate": [0.01, 0.03, 0.05, 0.07, 0.1], # 5 values
}
# 150 total combinations, only 15 sampled
```

**Missing critical XGBoost parameters:**
- `subsample` — prevents overfitting (currently hardcoded at 1.0 in walk_forward.py)
- `colsample_bytree` — feature sampling (currently hardcoded at 1.0)
- `min_child_weight` — regularization
- `gamma` — tree pruning
- `reg_alpha`, `reg_lambda` — L1/L2 regularization
- `scale_pos_weight` — class imbalance handling

**Fix**: Add at minimum `subsample: [0.7, 0.8, 0.9, 1.0]`, `colsample_bytree: [0.6, 0.7, 0.8, 0.9, 1.0]`, `min_child_weight: [1, 3, 5, 7]`. Increase search iterations to 50-100. Consider Optuna for Bayesian optimization instead of random search.

---

### 🔴 CRITICAL — Missing High-Value Features

Current feature set covers: offense (wRC+, wOBA, ISO, BABIP, K%, BB%), pitching (xFIP, xERA, K%, BB%, GB%, HR/FB%, velocity, entropy), defense (DRS, OAA, DefEff, framing), bullpen (pitch count, rest days, xFIP, IR%), baselines (Pythagorean, Log5).

**What's missing with high predictive value for F5:**

1. **Pitcher-vs-lineup L/R platoon splits** — Huge for F5 where the starter dominates
2. **Umpire strike zone tendencies** — Retrosheet client exists but is not wired into features
3. **Rest/travel features** — Days since last game, cross-timezone travel, day/night flip
4. **Streak/momentum** — Last 5/10 game win%, home/away specific
5. **Starter recent workload** — Pitch count last start, days rest, season pitch count trajectory
6. **First-inning tendencies** — Starter first-inning ERA/FIP (critical for F5)
7. **Historical H2H team performance** — Team vs team recent record

---

### 🟡 HIGH — Calibration Methodology Issues

`TODO.md` notes: "calibration hurt the 2024 holdout."

**Root causes:**
1. Only 10% calibration holdout (~1650 games) — small sample for isotonic regression
2. Platt scaling assumes probabilities are monotonically miscalibrated — may not hold
3. Calibration uses temporal tail of pre-holdout data, which may have different distribution

**Fix**:
- Try `identity` calibration (skip it) and compare — the stacking layer may already be well-calibrated
- Try `blend` method (average of isotonic + Platt)
- Increase calibration fraction to 15-20%
- Consider Venn-Abers calibration for better uncertainty estimates
- In walk-forward, the `calibration_fraction=0.15` is different from training's `0.10` — standardize

---

### 🟡 HIGH — Stacking Layer is Underutilized

**`stacking.py:40-44` — Only 3 meta-features:**
```python
DEFAULT_RAW_META_FEATURE_COLUMNS = (
    "home_team_f5_pythagorean_wp_30g",
    "home_team_log5_30g",
    "park_runs_factor",
)
```

The LR meta-learner sees XGBoost probability + 3 raw features. This severely limits the stacking layer's ability to correct XGBoost's mistakes.

**Fix**: Add more diverse meta-features:
- Away team baselines (currently only home team)
- Offensive differentials (home wRC+ minus away wRC+)
- Pitching quality differential
- Bullpen fatigue indicators
- Calibrated base rate (home team seasonal win%)

---

### 🟡 HIGH — No Feature Selection or Importance Tracking

All numeric columns with >1 unique value go into the model. No pruning of noisy/redundant features.

**Fix**:
- After training, compute SHAP values and drop features with near-zero importance
- Use recursive feature elimination or boruta-style selection
- Track feature importance across walk-forward windows to find stable predictors
- Drop features that are highly correlated (>0.95) with each other

---

### 🟡 HIGH — No Ensemble Diversity

Both F5 ML and F5 RL use XGBoost as the sole base learner. The stacking layer (LR) is lightweight but sees only one model's view.

**Fix**: Add LightGBM or CatBoost as alternative base learners. Stack across model types. Even a simple average of XGBoost + LightGBM typically outperforms either alone.

---

### 🟠 MEDIUM — Walk-Forward Configuration Not Experimented

**`walk_forward.py:53-67` — Hardcoded defaults:**
```python
DEFAULT_TRAIN_WINDOW_MONTHS = 6
DEFAULT_TEST_WINDOW_MONTHS = 1
DEFAULT_CALIBRATION_FRACTION = 0.15
DEFAULT_EDGE_THRESHOLD = 0.03
DEFAULT_MARKET_VIG = 0.04
DEFAULT_TIME_SERIES_SPLITS = 3
```

These are reasonable starting points but no evidence of systematic experimentation.

**Fix**: Run grid over `train_window_months=[3,4,6,9,12]`, `edge_threshold=[0.02,0.03,0.05]`, `market_vig=[0.03,0.04,0.05]`.

---

### 🟠 MEDIUM — F5 Run Line Model Includes Pushes

**`xgboost_trainer.py:43` — `"drop_ties": False` for RL**

F5 ties (pushes) are included in RL training but are uninformative noise — they represent games where neither side won/lost the bet.

**Fix**: Either `drop_ties=True` for RL too, or encode pushes as a separate class (3-class model) and handle separately.

---

### 🟠 MEDIUM — No Elo or Power Rating System

The baseline uses Pythagorean WP + Log5, which are solid but static per-window. An Elo rating that updates after every game would capture team quality changes more fluidly.

**Fix**: Implement a simple Elo system (or Glicko-2) that updates daily. Use Elo rating differential as a feature.

---

### 🟢 LOW — No Automated Experiment Tracking

Models get timestamped versions but no systematic A/B comparison infrastructure.

**Fix**: Add MLflow or even a simple experiment log (CSV/JSON) tracking: hyperparams, features used, holdout metrics, walk-forward aggregate stats.

### Key Findings from Experiment Data

#### 1. Offense Fix Improved the Base Model Consistently

Comparing #1 (Baseline) → #2 (Offense Fix) on the same 2024 holdout:
- **F5 ML AUC**: 0.5854 → **0.5908** (+0.54 pts)
- **F5 RL AUC**: 0.5861 → **0.5995** (+1.34 pts)
- **F5 ML Brier (base XGB)**: 0.2448 → **0.2441** (better)
- **F5 RL Brier (base XGB)**: 0.2152 → **0.2137** (better)

The offense fix delivered real improvement at the XGBoost level.

#### 2. Stacking Consistently Hurts F5 ML, Marginally Helps F5 RL

Across ALL 4 experiments:
- **F5 ML stacking Δ Brier**: -0.0003, -0.0002, -0.0002, -0.0008 → **always hurts**
- **F5 RL stacking Δ Brier**: +0.0003, +0.0002, +0.0004, +0.0005 → **always helps** (tiny)

The LR meta-learner with only 3 raw meta-features adds nothing for ML and barely helps RL. It's essentially passing through the XGBoost probability plus a slight Log5/Pythagorean blend.

#### 3. Platt Calibration Hurts Every Single Run

Across ALL experiments, calibration Δ Brier is negative:
- **F5 ML**: -0.0011, -0.0012, +0.00002, -0.0028 → **hurts or neutral**
- **F5 RL**: -0.0022, -0.0029, -0.0007, -0.0025 → **always hurts**

The Platt calibration step is **destructive**. The stacking layer already produces well-calibrated probabilities (ECE is 1.4%–3.4% pre-calibration). Adding Platt on top compresses the probability range into just 2 bins (visible in every reliability diagram — after calibration, all predictions land in bins 4-5, destroying the probability spread that existed before).

#### 4. Hyperparameter Search Always Lands on the Same Point

Every single run selected **`max_depth=6, n_estimators=100, learning_rate=0.01`**. This is the most conservative corner of the search space. The search space is too narrow and the RandomizedSearchCV with 15 iterations likely finds this early and can't explore further.

#### 5. 2025 Holdout Shows Degradation

Run #3 (2025 holdout) shows noticeably worse AUC than 2024 holdout runs:
- **F5 ML AUC**: 0.5708 vs 0.5908 (2024 offense fix)
- **F5 RL AUC**: 0.5647 vs 0.5995 (2024 offense fix)

Possible explanations: 2025 season may be structurally different, model may need more recent data emphasis, or features from the training window are less predictive forward. The 2025 holdout also **fails the reliability gap quality gate** (max gap 0.21 for ML, 0.23 for RL).

#### 6. Shorter Training Window (2023+) Slightly Beats Longer (2018+) on 2024

Comparing #2 (2018-2023 train) vs #4 (2023 + backfill to 2019):
- **F5 ML AUC**: 0.5908 vs **0.5936** (+0.28 pts for shorter)
- **F5 RL AUC**: 0.5995 vs 0.5893 (-1.02 pts for shorter)

Mixed signal, but suggests recent data may be more predictive for ML while RL benefits from more history.

#### 7. Walk-Forward ROI is Suspiciously High (+24.4%)

The 3-window Q2 2024 backtest shows +24.4% ROI on 1,013 bets. Key concerns:
- **Win rate**: 394W / 487L = 44.7% winners with average odds that must be >+130 equivalent
- **Market vig assumed at 4%**: Real market vig on F5 lines can be 5-8%
- **No real odds used**: The walk-forward generates synthetic odds from Log5 baselines, NOT from actual sportsbook lines — this massively overstates edge
- **Betting on nearly every game**: 1,013 bets across 1,207 scored games = 84% bet rate at 3% edge threshold
- **Sample size**: Only 3 windows × 1 month each — need at minimum 12+ windows for statistical significance

#### 8. Feature Importance Tells a Clear Story

Top features across ALL runs (remarkably stable):
1. **`home_team_log5_60g`** — Dominant feature (4-8% importance), team quality proxy
2. **`away_team_log5_60g`** / **`_30g`** — Same for away team
3. **`home_starter_k_pct_*`** — Starter strikeout rate across all windows
4. **`home_starter_xera_*`** — Expected ERA of home starter
5. **`away_starter_k_pct_*`** / **`xfip_*`** — Opposing starter quality
6. **`home_lineup_woba_7g`** / **`wrc_plus_7g`** — Short-window offense (only in offense-fix runs)

After the offense fix, lineup features appear in top-25, confirming the fix worked. The model is **overwhelmingly driven by team quality proxies (Log5) and starter quality (K%, xERA, xFIP)**. Bullpen, defense, weather, and park factors contribute minimally.

### Data-Driven Recommendations

1. **Drop Platt calibration immediately** — Use `identity` or raw stacking output. Calibration hurts in 7/8 measurements.

2. **Reconsider stacking for F5 ML** — The LR layer consistently worsens F5 ML. Either add more meta-features or skip stacking for ML entirely.

3. **Expand hyperparameter search** — Every run converges to `d6/n100/lr0.01`. Add `subsample`, `colsample_bytree`, `min_child_weight`, and increase search iterations to 50+. The walk-forward uses different params (`d3/n120/lr0.05`) — these differences are untested against each other.

4. **Don't trust the walk-forward ROI** — +24.4% is based on synthetic odds, not actual sportsbook lines. Need to run against historical Odds API data.

5. **Test training window sizes systematically** — Run holdout protocol with `start_year` in {2019, 2021, 2022, 2023} to find optimal history depth.

6. **Prune low-importance features** — 224 features but importance is heavily concentrated in ~20. Weather, ABS, adjusted framing, many short-window defense metrics contribute near-zero. Dropping them may reduce overfitting.

7. **Add cross-team differential features to stacking** — Instead of just `home_team_log5_30g`, add differentials like `home_starter_k_pct_30s - away_starter_k_pct_30s`.

8. **Run longer walk-forward** — Extend to April-September 2024 (6 windows minimum) for more robust evaluation.
