# MLBPrediction2026 Handoff

## Goal

The current direction is:

- use the local scraper database as the primary live odds source
- keep all historical odds snapshots locally for backtesting and CLV work
- build and refresh a cached daily slate in SQLite
- have the dashboard read cached slate results instead of rebuilding everything on every page load
- move toward the new run-count model family as the main prediction architecture

## What Is Already Implemented

### Live odds architecture

- The dashboard/pipeline now prefers the local scraper database at:
  - `OddsScraper/data/mlb_odds.db`
- The pipeline falls back to cached/API odds only if scraper data is missing.
- Odds API key pooling and monthly exhaustion tracking are implemented.
- Local odds caching with TTL exists for non-scraper fallback paths.

### Cached slate behavior

- `/api/slate` now reads from a cached SQLite payload by default.
- A fresh slate rebuild only happens when:
  - there is no cached payload for the target day, or
  - the route is called with `refresh=true`
- Cached payloads are stored in:
  - `cached_slate_responses`

### Game completion tagging

- Each game in the slate payload now includes:
  - `game_status`
  - `is_completed`
- These fields are also persisted in:
  - `daily_pipeline_results`
- This is the current split between:
  - historical/final games
  - live/upcoming games

### Model direction

- Legacy models still exist for benchmark/reference:
  - F5 ML legacy classifier
  - F5 RL legacy classifier
- New run-count models are wired into prediction:
  - `f5_home_runs_model`
  - `f5_away_runs_model`
  - `full_game_home_runs_model`
  - `full_game_away_runs_model`
- F5 and full-game projected runs are now part of the prediction payload.
- F5 ML and F5 RL probability generation now use run-count-derived pricing where available.

### Dashboard behavior

- Dashboard emphasizes:
  - `bet365`
  - market average
- Game cards include collapsible sections for:
  - model read
  - full game market
  - inputs
- Value play threshold is currently `6%`.

## What Still Needs To Be Set Up

### 1. Always-on refresh job

The intended production flow is:

1. scrape odds
2. refresh cached slate
3. dashboard reads cache only

This should be run on an always-on machine.

Best near-term choice:

- MacBook or other always-on host

Suggested cadence:

- every 60 minutes

Important:

- the slate refresh should run only after a scraper refresh, not independently

### 2. Lightweight live/upcoming scraper runner

The current scraper runners are mostly backfill-oriented:

- `OddsScraper/run_2026.py`
- `OddsScraper/run_2025_2026.py`
- `OddsScraper/run_full.py`

For automation, a lightweight runner is still needed.

That runner should scrape only:

- today
- tomorrow
- optionally the next 7 days

That keeps live refreshes fast and preserves opening-to-closing movement for upcoming games.

### 3. Scheduler setup

Once the lightweight runner exists, create one scheduled job that does:

1. run scraper
2. run daily pipeline refresh
3. log output

Use one of:

- `launchd` on macOS
- `cron` on Linux / Raspberry Pi / VPS
- Windows Task Scheduler on Windows

### 4. Optional cloud move

If moved off local machines, use an always-on host.

Recommended shape:

- small VPS
- SSH access
- local SQLite on disk
- scheduled scraper + slate refresh jobs

Avoid using Vercel for the scraper/database layer. It is fine for frontend/API hosting, but not a good fit for Playwright scraping plus persistent SQLite.

## Manual Commands

### Backend

```powershell
.\.venv\Scripts\python.exe -m uvicorn src.dashboard.main:app --reload --port 8010
```

### Frontend

```powershell
Set-Location dashboard
npm run dev
```

### Force a fresh slate rebuild

Open:

- `http://127.0.0.1:8010/api/slate?refresh=true`

### Daily pipeline CLI

```powershell
.\.venv\Scripts\python.exe -m src.pipeline.daily --date today --mode prod --dry-run --db-path data\mlb.db
```
